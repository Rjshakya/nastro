# @nastro-dev/notion-orm-cli

CLI for managing Notion databases using `@nastro-dev/notion-orm`. Define your schema in TypeScript, push it to Notion, and keep your database structure in version control.

## Installation

### Global install

```bash
npm install -g @nastro-dev/notion-orm-cli
```

### Or use with npx (no install)

```bash
npx @nastro-dev/notion-orm-cli push
```

## Quick Start

### 1. Create a config file

Create `notion-orm.config.ts` in your project root:

```typescript
export default {
  token: process.env.NOTION_TOKEN!,     // Notion integration token
  rootPage: "your-root-page-id",        // Notion page where databases live
  schema: "src/schemas/**/*.ts",        // glob pattern for schema files
};
```

### 2. Define your schema

Create `src/schemas/tasks.ts`:

```typescript
import { table, title, richText, select, date, checkbox } from "@nastro-dev/notion-orm";

export const tasksTable = table("Tasks", {
  name: title(),
  description: richText(),
  status: select({
    options: ["Todo", "In Progress", "Done"],
  }),
  dueDate: date(),
  completed: checkbox(),
});
```

### 3. Push to Notion

```bash
notion-orm push
```

This will:
- Load all schema files matching your glob pattern
- Compare each table against the existing Notion database
- Create new databases for tables that don't exist
- Update existing databases when properties are added or changed
- Generate `notion-orm.generated.ts` with database ID mappings

## Commands

### `push`

Push schema changes to Notion.

```bash
notion-orm push
notion-orm push --force                    # allow property deletions
notion-orm push --rename "oldName=newName" # rename properties
```

#### Options

| Flag | Description |
|------|-------------|
| `-f, --force` | Force push even if properties would be deleted |
| `-r, --rename <renames...>` | Rename properties in format `"oldName=newName"`. Can be used multiple times. |
| `-c, --config <path>` | Path to config file |
| `-t, --token <token>` | Notion integration token (overrides config) |
| `--root-page <id>` | Notion root page ID (overrides config) |
| `-s, --schema <path>` | Schema file glob pattern (overrides config) |

#### Change Detection

The CLI detects four types of changes:

- **Added** (`+`) — New properties in your schema
- **Removed** (`-`) — Properties missing from your schema
- **Type changed** (`~`) — Property type mismatch
- **Renamed** (`→`) — Properties renamed via `--rename`

By default, pushes with property deletions are **blocked** to prevent accidental data loss. Use `--force` to override.

#### Renaming Properties

```bash
notion-orm push --rename "Description=Notes" --rename "Due Date=Deadline"
```

This converts a `remove` + `add` pair into a `rename` operation, preserving existing data.

### Global Options

| Flag | Description |
|------|-------------|
| `-c, --config <path>` | Path to configuration file |
| `-t, --token <token>` | Notion integration token |
| `--root-page <id>` | Notion root page ID |

## Configuration

The CLI looks for `notion-orm.config.ts`, `notion-orm.config.js`, or `notion-orm.config.mjs` in the current directory.

### Config file format

```typescript
export interface NotionOrmConfig {
  token: string;      // Notion integration token
  rootPage: string;   // Root Notion page ID
  schema: string;     // Glob pattern for schema files
}
```

### Environment variables

The CLI automatically loads `.env` files. You can use `NOTION_TOKEN` as a fallback:

```bash
NOTION_TOKEN=secret_xxx notion-orm push
```

### CLI overrides

CLI flags take precedence over config files:

```bash
notion-orm push -t secret_xxx --root-page abc123 -s "src/db/*.ts"
```

## Generated Files

After a successful push, the CLI creates `notion-orm.generated.ts`:

```typescript
// Auto-generated by notion-orm push
// Do not edit manually

export const databaseMapping = {
  "Tasks": "database-id-1",
  "Projects": "database-id-2",
  // table title → database ID
  // database ID → data source ID
} as const;
```

This mapping is used by the ORM runtime to resolve table names to Notion database IDs.
